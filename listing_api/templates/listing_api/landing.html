What one request returns

Three things in one round-trip

No stitching together four vendors. Metadata, copy, and embeddings — all from the same understanding of the garment.

1. Structured metadata

Category, subcategory, silhouette, primary + secondary colors, pattern, material, brand (normalized), size label, condition grade, era estimate, style tags. Every enum field is drawn from a controlled vocabulary — no synonym sprawl.

2. Listing copy

Title (SEO-length), 3–5 sentence description in resale-house style, and 15–25 search-optimized tags. Factual only — never invents details the photos don't support.

3. Embeddings

Pre-computed BGE-base text embedding (768d) and CLIP ViT-B/32 image embedding (512d). Drop into pgvector, Pinecone, or Weaviate for "find similar listings" and visual search.

One request

POST multipart, get JSON back

curl -X POST https://theoracle.up.railway.app/listing/api/v1/analyze/ \
  -H "Authorization: Bearer sk_live_..." \
  -F "images=@front.jpg"  -F "images=@label.jpg" -F "images=@detail.jpg" \
  -F "roles=front,label,detail" \
  -F "hint=Saint Laurent oversized wool coat"

Why it's different

Controlled vocabularies, not synonym soup

Most vision-tagging tools free-generate labels. You end up with "ivory", "cream", "off-white", and "winter white" for the same garment. That breaks filters and faceted search. Every enum field here is drawn from a fixed, editable taxonomy — so your catalog stays clean while embeddings handle the nuance.
